Parameter support varies by endpoint and model.

## Common request parameters

| Parameter | Type | Typical usage |
| --- | --- | --- |
| `model` | string | Required model id (for example `openai/gpt-5-nano`). |
| `stream` | boolean | Enable SSE token streaming on text endpoints. |
| `temperature` | number | Control output randomness when supported. |
| `top_p` | number | Nucleus sampling control where supported. |
| `max_tokens` / `max_output_tokens` | integer | Cap output length and cost. |
| `stop` | string or string[] | Define explicit stop sequences. |
| `tools`, `tool_choice` | object / string | Tool-calling controls on compatible endpoints. |
| `response_format` | string or object | Ask for plain text, JSON, or schema-constrained responses. |
| `service_tier` | string | Request a supported pricing or routing tier such as `priority` or `flex` on compatible text APIs. |
| `meta`, `usage` | boolean | Include extra metadata and usage details in responses. |

`service_tier` is supported on the main text request surfaces:

- [Anthropic Messages](./endpoint/anthropic-messages.mdx)
- [Chat Completions](./endpoint/chat-completions.mdx)
- [Responses](./endpoint/responses.mdx)

Accepted values are `standard`, `priority`, `flex`, and `batch`. For synchronous text requests, `batch` returns a validation error with a pointer to the Batch API because batch pricing applies to deferred batch execution, not normal request/response calls.

AI Stats maps these gateway-normalized tier values to provider-native controls internally, such as OpenAI or Anthropic provider-specific fields, so callers can use the same `service_tier` values across supported text surfaces.

## Provider routing parameters

The `provider` object lets you influence routing behavior:

| Field | Type | Purpose |
| --- | --- | --- |
| `order` | string[] | Preferred provider order. |
| `only` | string[] | Restrict routing to specific providers. |
| `ignore` | string[] | Exclude specific providers. |
| `include_alpha` | boolean | Allow alpha providers in routing decisions. |
| `sort` | string or object | Rank providers for this request, typically by `price`, `latency`, or `throughput`. |
| `required_execution_region` | string | Restrict routing to providers that execute in a required region. |
| `required_data_region` | string | Restrict routing to providers that store or process data in a required region. |
| `require_zero_data_retention` | boolean | Restrict routing to providers that meet zero-data-retention requirements. |
| `max_price` | object | Set price ceilings for prompt, completion, image, audio, or request costs. |
| `quantizations` | string[] | Narrow routing to specific quantization variants when supported. |

## Provider-specific options

Use `provider_options` when you need provider-native configuration such as:

- `openai.context_management`
- `openai.prompt_cache_retention`
- `anthropic.cache_control`
- `google.cache_control`
- `google.cached_content`

Keep these isolated so you can preserve portability between providers.

## Debug parameters

Use `debug` for controlled troubleshooting:

| Field | Type | Purpose |
| --- | --- | --- |
| `enabled` | boolean | Enable debug mode for the request. |
| `return_upstream_request` | boolean | Include transformed upstream request payload. |
| `return_upstream_response` | boolean | Include upstream response payload where available. |
| `trace` | boolean | Return routing/debug traces. |
| `trace_level` | `summary` or `full` | Control trace verbosity. |

`debug` data may contain sensitive request context. Use only in development or controlled environments.

## Example

```json
{
  "model": "openai/gpt-5-nano",
  "input": "Summarize this changelog.",
  "stream": false,
  "temperature": 0.3,
  "max_output_tokens": 300,
  "provider": {
    "order": ["openai", "anthropic"],
    "ignore": ["some-provider"],
    "sort": "latency",
    "required_execution_region": "eu",
    "require_zero_data_retention": true
  },
  "debug": {
    "enabled": true,
    "trace": true,
    "trace_level": "summary"
  }
}
```
